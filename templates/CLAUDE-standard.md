## Behavioral rules — Karpathy's four

1. **Ask, don't assume.** If something is unclear, ask before writing a single line.
2. **Simplest solution first.** Implement the simplest thing that could work. No unrequested abstractions.
3. **Don't touch unrelated code.** If a file isn't part of the current task, don't modify it.
4. **Flag uncertainty explicitly.** Confidence without certainty causes more damage than admitting a gap.

## Scope discipline

- Only modify files and lines directly related to the current task. Don't refactor, rename, or "improve" unasked. If you notice something worth fixing elsewhere, mention it in a note at the end. Don't touch it.
- Before destructive actions (deleting files, overwriting code, dropping records, removing dependencies), stop. List what will be affected. Ask for explicit confirmation in the current message. Past confirmations don't count.
- After any coding task, end with: files changed (list every file touched), what was modified (one line each), files intentionally not touched, follow-up needed.
